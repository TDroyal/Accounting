# 需求报告

> 文档编号：01
> 项目名称：Accounting（个人记账应用）
> 文档版本：v1.0
> 更新日期：2026-07-12

---

## 1. 项目背景

开发者是一名互联网大厂程序员，未来几年内的核心生活目标是**购买一套属于自己的房子**。买房需要积累可观的首付款，而当前最直接的障碍是：

- **缺乏记账习惯**：日常开销零散、无记录，月底说不清钱花到了哪里；
- **无复盘依据**：缺少结构化的开销数据，无法判断哪些支出可以压缩、能省下多少；
- **缺乏量化目标**：没有月度/年度预算与实际开销对比，难以形成省钱节奏。

因此需要一款**便捷的个人记账应用**，让记录开销的门槛降到最低，并提供日、月、年多粒度的可视化统计，支撑「记录 → 复盘 → 调整 → 省钱」的闭环。

## 2. 项目目标

### 2.1 业务目标
| 目标 | 说明 | 衡量指标 |
|------|------|----------|
| 降低记账门槛 | 3 步内完成一笔记录，支持手机随手记 | 单笔记录耗时 ≤ 15s |
| 多粒度可视化复盘 | 日/月/年统计与图表 | 覆盖 3 个时间粒度、≥4 种图表 |
| 支撑省钱决策 | 分类占比、趋势对比、预算控制 | 支出分类可识别 Top 3 |
| 数据归属自己 | 单用户、数据可导出 | 支持导出 CSV/JSON |

### 2.2 用户体验目标
- **手机端**：随时随地快速记账，首屏即记账入口；
- **电脑端**：大屏做月度/年度复盘，图表清晰、信息密度高；
- **响应式**：同一套代码适配两端，布局自动切换。

## 3. 用户画像

| 维度 | 描述 |
|------|------|
| 角色 | 互联网大厂程序员（单一主要用户） |
| 记账习惯 | 目前无记账习惯，需要低门槛引导 |
| 设备 | 手机（iOS/Android 浏览器）+ 电脑（Mac 浏览器） |
| 使用场景 | 手机：消费后即时记账；电脑：周末/月末复盘 |
| 技术能力 | 高，可接受无原生 App，使用浏览器即可 |
| 核心诉求 | 快速记录 + 看清钱花在哪 + 知道能否省下来 |

## 4. 功能需求

### 4.1 核心功能模块

#### 4.1.1 账户与登录（单用户）
- 注册 / 登录 / 修改密码
- JWT 鉴权，会话持久化（Redis）
- 数据按用户隔离

#### 4.1.2 记账录入（核心）
- 录入字段：金额、分类、发生时间（默认当前）、备注、类型（支出/转账）
- 快捷记账：默认当前时间、常用分类置顶、数字键盘优先
- 转账：记录账户间资金流动（不计入支出统计）

#### 4.1.3 支出分类（核心细分领域）
预置分类，支持增删改，结构为**分类树**（一级分类 + 二级分类，便于扩展）：

| 一级分类 | 二级分类（示例） | 说明 |
|----------|------------------|------|
| 交通 | 地铁、公交、打车、加油、停车 | 通勤与出行 |
| 话费 | 手机、宽带 | 通讯 |
| 转账 | 还款、红包、AA、其他 | 资金转移，不计入支出 |
| 外卖 | 美团、饿了么 | 外卖订餐 |
| 吃饭 | 早餐、午餐、晚餐、聚餐 | 到店/自做之外的就餐 |
| 房租 | 租金、水电、物业 | 居住成本 |
| 购物 | 日用、服饰、数码 | —— |
| 娱乐 | 影音、游戏 | —— |
| 其他 | —— | 兜底分类 |

> 转账为「资金转移」性质，不计入支出总额，但在流水中可查。

#### 4.1.4 账户管理（支撑转账，可选优先实现）
- 账户：现金、银行卡、支付宝、微信、信用卡等
- 记录各账户余额，转账时联动扣减/增加

#### 4.1.5 统计与可视化（核心）
按 **日 / 月 / 年** 三种粒度提供：

| 粒度 | 展示内容 | 图表类型 |
|------|----------|----------|
| 日 | 当日支出总额、分类占比、明细列表 | 饼图 + 列表 |
| 月 | 当月累计、每日趋势、分类占比、与上月对比 | 折线 + 柱状 + 饼图 |
| 年 | 年度累计、月度趋势、Top 分类、月均支出 | 折线 + 柱状 + 排行 |

详见 [07-可视化与统计设计.md](./07-可视化与统计设计.md)。

#### 4.1.6 预算与目标（辅助省钱）
- 设置月度预算上限，统计页显示「已用/剩余/超支」
- （可选）年度省钱目标进度

#### 4.1.7 数据导出
- 导出明细为 CSV / JSON，便于本地留存或二次分析

### 4.2 功能优先级（MVP → V2）

| 优先级 | 模块 | 说明 |
|--------|------|------|
| P0（MVP） | 登录、记账、分类、月/日统计 | 上线可用 |
| P1 | 年度统计、预算、导出 | 增强复盘 |
| P2 | 账户管理、转账联动余额、省钱目标 | 完整体验 |

## 5. 非功能需求

| 类别 | 要求 |
|------|------|
| 响应式 | 手机(<768px)/平板/电脑(≥1024px) 三档断点，核心操作手机端可用 |
| 性能 | 统计接口 P95 ≤ 500ms（依赖 Redis 缓存预聚合） |
| 安全 | 密码加盐哈希(bcrypt)、JWT 鉴权、HTTPS、SQL 参数化 |
| 可用性 | 单笔记账 ≤ 15s；图表移动端可读 |
| 数据 | 每日可导出；统计结果可缓存重建 |
| 可维护 | 前后端分离、Monorepo、文档与代码同步 |
| 部署 | 单机 Docker Compose 即可运行 |

## 6. 约束与假设

- **约束**：
  - 技术栈固定：前端 Vue 3，后端 Gin，数据库 MySQL + Redis；
  - 纯 Web SPA，不开发原生 App；
  - 单用户体系，不做多用户/家庭账本；
  - Monorepo 单仓管理前后端。
- **假设**：
  - 用户仅一人，并发量极低，无需复杂分布式；
  - 部署在一台自有/云服务器，域名 + HTTPS 由反向代理(Nginx)终止。

## 7. 术语表

| 术语 | 含义 |
|------|------|
| 流水 / Transaction | 一笔记账记录（支出或转账） |
| 分类树 | 一级分类 + 二级分类的层级结构 |
| 转账 | 账户间资金转移，不计入支出统计 |
| 预聚合 | 在写入时或定时任务中提前计算统计结果并缓存，提升查询性能 |
